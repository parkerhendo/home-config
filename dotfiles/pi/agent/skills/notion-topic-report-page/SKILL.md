# Notion Topic Report Page

## Purpose

For every Braintrust Topics report, create a **new Notion child page** instead of appending the report to an existing page. The page must contain:

1. The user-facing analyst report.
2. A collapsible debug/provenance section with relevant metadata: data windows, topic-map/report IDs, commands or queries used, verified traces, skipped traces, and **Pi session/turn** metadata such as model, cost, tool calls, and turns when available.

## Quick start

1. Find the parent page:
	- Search Notion for `Loop Project Topics Reports`.
	- Use the exact parent page if found.
	- If multiple plausible pages exist, ask the user which one to use.
2. Create a new child page under that parent.
3. Title format:
	- Always include the Braintrust project name in the page title.
	- `{Project name} — {Facet} topics report — {YYYY-MM-DD}`
	- If comparing days: `{Project name} — {Facet} topics report — {YYYY-MM-DD} vs {YYYY-MM-DD}`
	- If a same-title page already exists, append a short qualifier like `(verified)` or timestamp.
4. Paste the analyst-facing report exactly as generated by the report model.
5. Add only a collapsed debug/provenance toggle after the report.

## Page structure

Use Notion-flavored Markdown, but **do not impose a wrapper structure on the analyst-facing report**.

Paste the report body exactly as generated by the report model:

- Do not add a `Date`, `Summary`, `Topic movement`, `Traces worth manual investigation`, or any other visible section unless it is already present in the generated report.
- Do not rewrite headings, split/merge sections, reorder bullets, add tables, or move counts into the visible report.
- Do not promote debug-only movement counts into the public report. If topic movement/counts are useful provenance, put them only inside the collapsed debug section.
- Preserve the report's existing compact structure and wording, except for mechanical Notion Markdown escaping required for correct rendering.

After the report body, append exactly one collapsed debug/provenance section:

```md
<details>
<summary>Debug and provenance</summary>
	### Data source
	- Org: ...
	- Project: ...
	- Object ID: ...
	- Facet/topic map: ...
	- Topic map function ID: ...
	- Topic map version/report created at: ...
	- Compared windows: ...

	### Pi session / turn metadata
	- Pi model used for this analysis: ... or Not captured
	- Pi estimated cost for this turn/session: ... or Not captured
	- Pi turns included in this report generation: ... or Not captured
	- Pi tool calls made for this report: ...
	- CLI/API commands used: ...

	### Trace handling
	- In-progress traces skipped: ...
	- Verified highlighted traces: ...
	- Candidate traces dropped after inspection: ...

	### Notes
	- Any topic-label quality caveats or data limitations.
</details>
```

## Debug metadata checklist

Include these when available; do not fabricate missing values.

- **Report provenance:** `bt topics config` output details, topic map function ID, report version, report `created_at`, cluster run settings.
- **Data windows:** date/time windows compared, cutoff time, total processed/classified rows, skipped `no_match`/`skipped` counts if used.
- **Topic counts:** before/after counts, deltas, whether counts are full-day or same-cutoff comparisons.
- **Trace verification:** each highlighted trace URL, verification method, and whether the built-in `thread` preprocessor was used.
- **Skipped traces:** any trace excluded because it had running/in-progress spans or was incomplete.
- **Pi session / turn metadata:** model, cost, token usage, turn count, and tool-call count must come from the current Pi session/turn that generated the report, not from the Braintrust traces being analyzed. If Pi does not expose a field, write `Not captured in this Pi session`.
- **Tooling:** important `bt` CLI commands, Notion page creation target, scripts used, and any failures/retries.

## Rules

- Never overwrite an existing Topic Report page unless the user explicitly asks.
- Do not include private chain-of-thought. Debug sections should be operational provenance, not hidden reasoning.
- If a highlighted trace was not inspected, do not present it as verified.
- If Pi cost/model/tool-call/turn metadata is unavailable, say so rather than guessing or substituting metadata from the analyzed traces.
- Keep the public report concise and actionable; put bulky raw debug details inside the toggle.
- After creating the page, return the Notion URL and a one-line summary of what was saved.
